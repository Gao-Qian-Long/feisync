# 插件测试计划

## 测试目标

确保插件核心功能正常工作，包括：

1. 设置界面正确加载和保存配置。
2. 飞书认证模块能够获取有效的 `tenant_access_token`。
3. 文件夹管理模块能正确创建和查找文件夹。
4. 文件上传模块能将本地文件上传到飞书Drive。
5. 文件监控器能正确监听文件变化并触发同步。
6. 同步引擎能协调各模块完成一次完整的同步。

## 测试环境

- Obsidian 桌面端（最新版本）。
- 一个测试用的飞书开放平台应用（Sandbox），拥有Drive API权限。
- 一个本地Obsidian仓库，包含测试文件夹。

## 测试类型

### 1. 单元测试

使用 Jest 或类似的测试框架，对独立的类进行测试。

**待测试的模块：**

- `FeishuAuthManager`：模拟网络请求，测试令牌获取和刷新逻辑。
- `FeishuFolderManager`：模拟API响应，测试文件夹路径确保逻辑。
- `FeishuMediaUploader`：模拟上传请求，测试文件大小检查和上传流程。
- `FileWatcher`：模拟Vault事件，测试事件过滤和防抖调度。

**Mock 策略：**

- 使用 `jest.mock` 替换 `fetch` 调用。
- 模拟 Obsidian API（`Vault`, `Plugin` 等）可以使用自定义的模拟对象。

### 2. 集成测试

在真实 Obsidian 环境中测试插件，但使用模拟的飞书API（例如使用本地 mock server）。验证插件设置界面、命令注册、ribbon 按钮等。

### 3. 端到端测试（手动）

使用真实的飞书应用凭证，在真实的 Obsidian 仓库中执行同步操作，检查飞书Drive中是否出现对应文件。

## 测试用例

### 认证模块

- **TC1**：使用正确的 App ID 和 App Secret 应成功获取令牌。
- **TC2**：使用错误的凭证应抛出适当的错误。
- **TC3**：令牌过期后应自动刷新。

### 文件夹管理

- **TC4**：确保文件夹路径 `a/b/c` 能逐级创建文件夹。
- **TC5**：如果文件夹已存在，应直接返回其 token（避免重复创建）。
- **TC6**：父文件夹 token 无效时应报错。

### 文件上传

- **TC7**：上传一个小文本文件（<1MB）应返回 file_token。
- **TC8**：上传超过 20MB 的文件应报错。
- **TC9**：上传同名文件应覆盖（先删除后上传）。

### 文件监控

- **TC10**：在监控文件夹内创建新文件，应触发同步（防抖后）。
- **TC11**：修改文件应触发同步。
- **TC12**：删除文件应触发同步（如果配置了删除同步）。
- **TC13**：监控文件夹外的文件变化不应触发同步。

### 同步引擎

- **TC14**：手动执行同步命令，应上传所有本地文件。
- **TC15**：同步过程中网络中断，应记录错误并允许重试。
- **TC16**：同步完成后应显示成功通知。

## 测试工具

- **Jest**：用于单元测试。
- **Obsidian Plugin Tester**（如果存在）：用于集成测试。
- **Postman** 或 **curl**：验证飞书API端点。
- **Fiddler** / **Charles**：抓包调试。

## 测试数据

- 准备一个测试用的飞书应用，获取 `app_id` 和 `app_secret`（可存储在环境变量中，不提交到代码库）。
- 准备一个测试文件夹，包含多种类型的文件（`.md`, `.txt`, `.png` 等）。

## 持续集成

在 GitHub Actions 中配置自动化测试流程：

1. 安装 Node.js 和依赖。
2. 运行单元测试（使用模拟，不调用真实API）。
3. 构建插件（`npm run build`）确保无编译错误。

## 风险与缓解

- **飞书API变动**：可能导致测试失败。定期更新API调用方式。
- **网络不稳定**：集成测试可能因网络问题失败。增加重试机制，或使用模拟。
- **Obsidian 版本升级**：API 可能变化。测试时锁定 Obsidian 版本范围。
